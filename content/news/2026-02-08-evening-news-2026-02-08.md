### AI Code Editor活用の実際

Cameron氏（10年以上のエンジニア経験）の証言：

> *「AI code editorは間違いなく最大のアドバンテージだった。10年の経験があったからこそ、AIツールを適切に活用できた。アーキテクチャを正しく設計し、AIの出力をレビューし、ミスを素早く発見する能力と、AIツールの組み合わせが、数年前には不可能だった方法でソロ開発を可能にした。」*

2. **スピード重視**: 完璧より速度、早期ユーザーフィードバック
3. **実用主義**: オーバーエンジニアリングを回避

## 「エージェンティック・コーディングは生産性を下げる」— Haskell界の重鎮が衝撃検証

一方で、**Gabriel Gonzalez氏（Haskellの著名開発者）**が発表した論考「Beyond agentic coding」が、AI開発界に波紋を広げています。

### エージェンティック・コーディングの3つの問題

**Hacker News で95点獲得**した彼の主張：

#### 1. **実証的な証拠**: 生産性向上なし
- **Becker研究**: アイドル時間が約2倍に増加
- **Shen研究**: 固定的な成果で測定すると、むしろパフォーマンス低下
- **面接体験**: エージェンティック・ツール使用者の方が成績が悪い傾向

#### 2. **Flow State の破壊**
現在のチャット型AIエージェントの問題：
- **高い注意要求**: 待機時間中も中断可能状態を維持
- **間接的インターフェース**: コードより会話に時間を使う
- **平静の破壊**: 常に刺激が必要、受動的な学習ができない

#### 3. **Calm Technology の欠如**
優れたツールの特徴：
- **注意への要求を最小化**
- **Pass-through設計**: ツールでなく作業対象に集中
- **平静の創造**: フロー状態の維持・促進

### 代替案: 「Calm Technology」によるAI支援

Gabriel氏の提案する次世代AI開発ツール：

**1. Facet-based Project Navigation**
- セマンティック構造でプロジェクトを整理
- 「String interpolation regression」vs「dhall/tests/format/issue2078A.dhall」
- 意図でコードを探索、理解を深める

**2. Automated Commit Refactor**
- 大きなコミットを自動的に論理的な小コミットに分割
- 人間のレビュー負荷を削減（現在のエージェント型は増加）

**3. File Lens**
- **「Focus on...」**: 特定機能のみ表示
- **「Edit as...」**: 他言語として編集（Python→Haskell等）

**結論**: *「チャットは LLM への最も興味の薄いインターフェースだ」*

## 夕刊の洞察

### **AI Tool選択の新しい原則**

今夜の3つの事例から見える重要なパターン：

#### 1. **経験 + AI = 指数的効果**
Cameron Trew氏の成功は「10年の経験 × AI Code Editor」の組み合わせ。AIは初心者の万能薬ではなく、**熟練者の加速装置**として最も威力を発揮。

#### 2. **Flow State優先の設計思想**
Gabriel Gonzalez氏の指摘通り、「AIが凄いかどうか」より「人間がフロー状態を維持できるか」が重要。チャット型は注意を散漫にし、インライン型は作業に集中させる。

#### 3. **ローカル vs クラウドの戦略的選択**
LocalGPTのような完全ローカル実行は、プライバシーとカスタマイズ性を重視する開発者にとって新しい選択肢。

### **実践的な選択指針**

- **速度重視**: Claude Code + 標準スタックで高速プロトタイピング
- **フロー重視**: インライン提案型を選び、チャット型は最小限に
- **プライバシー重視**: ローカル実行可能なツールを検討
- **経験活用**: AIは補完ツール、アーキテクチャ設計は人間が主導

**2026年2月8日の結論**: AIツール選択は「機能の多さ」ではなく「作業フローへの適合性」で決まる。Cameron Trew氏の$82k MRR達成は、この原則の見事な実証例です。
