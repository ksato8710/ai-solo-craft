## 🔄 フィードバックループ設計（最重要）

### 原則：「使うたび自律的に品質や仕組みが強くなる」

```
┌─────────────────────────────────────────────────────────────┐
│                    自律改善サイクル                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ① 運用実行          ② 問題検知          ③ 原因分析       │
│   ─────────────  →  ─────────────  →  ─────────────        │
│   cron/手動実行       エラーログ          失敗パターン        │
│   記事作成            CI失敗              抽出               │
│   デプロイ            URL 404                                │
│                                                             │
│         ↑                                    ↓              │
│                                                             │
│   ⑥ 次回自動適用      ⑤ 仕組み化          ④ 修正案         │
│   ─────────────  ←  ─────────────  ←  ─────────────        │
│   Skill更新が         husky/CI追加        MEMORY.md記録      │
│   自動反映            cronプロンプト更新  Skill修正          │
│                       チェックリスト追加                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 具体的な実装

#### ループ1: 失敗パターン → 仕組み化（既存強化）

**現状:** MEMORY.mdに手動記録 → 手動でスキル更新
**改善:** 

```yaml
# 週次cron: 失敗パターン分析
name: feedback-loop-analysis
schedule: "0 9 * * 1"  # 毎週月曜 9:00
prompt: |
  MEMORY.mdの「失敗パターン」と「成功体験ログ」を分析し:
  1. 過去1週間に追加された項目を抽出
  2. 仕組み化されていないものを特定
  3. 仕組み化の提案（husky/CI/Skill/cronのいずれか）を作成
  4. #tifa に報告
```

#### ループ2: 品質チェック → 自動修復

**現状:** check:images がエラーを検知 → 手動修正
**改善:**

```yaml
# 自動修復Skill: image-auto-fix
trigger: check:images 失敗時
action: |
  1. 未設定画像を検出
  2. 記事テーマから適切なUnsplash画像を選定
  3. frontmatter自動更新
  4. 再チェック実行
  5. 成功なら自動コミット
```

#### ループ3: 競合リサーチ → 記事ネタ

**新規:**

```yaml
# 週次cron: 競合AIメディア分析
name: competitor-research
schedule: "0 10 * * 3"  # 毎週水曜 10:00
prompt: |
  以下のAIメディアの直近1週間の記事を分析:
  - TLDR AI
  - The Rundown AI
  - AI Breakfast
  - Ben's Bites
  
  Output:
  1. 話題のトピック（週間トレンド）
  2. 我々がカバーしていないテーマ
  3. 記事アイデア3つ
  4. Content Studio に登録
```

#### ループ4: UI品質 → 自動監査

**新規:**

```yaml
# Skill: ui-consistency-audit
trigger: コンポーネント変更時
action: |
  ui-design-system.md の原則に照らして監査:
  1. border-radius が 16-20px か
  2. グラデーション背景を使っているか
  3. insetハイライトがあるか
  4. 違反があれば修正候補を提示
```

## 🧠 設計原則（元記事からの学び）

### 1. Skillは「自動で発火する」

> 「プロンプトライブラリは誰も見ない。Skillは関連ファイルを検知して自動発火する。この差がすべて」

**適用:** 全Skillに明確な `trigger` 条件を定義。手動呼び出し前提のSkillは廃止候補。

### 2. 監査は継続的に

> 「年1回のアクセシビリティ監査ではなく、毎コミットの自動チェック」

**適用:** check:images のパターンを全品質指標に拡大（UI一貫性、リンク整合性、パフォーマンス）

### 3. バイパスパターンは宝

> 「ルール違反は批判材料ではなく、仕組みの不備を示すシグナル」

**適用:** 失敗パターン → 仕組み化 のループを自動化。MEMORY.mdを「学習データセット」として扱う。

### 4. MCPでリアルタイム接続

> 「コピペしたトークン値は即座に古くなる。MCPで常に最新データを参照」

**適用:** Supabase MCP, GitHub MCP の導入で、DB/コード状態をリアルタイム参照可能に。
