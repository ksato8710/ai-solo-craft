# 個別記事ワークフロー（dev-knowledge / case-study）

朝刊・夕刊 Digest とは別の、深掘り記事向けフロー。

> **全体像:** `docs/operations/WORKFLOW-OVERVIEW.md`
> **品質チェック:** `docs/operations/CHECKLIST.md`
> **taxonomy:** `specs/content-policy/spec.md`

---

## 概要

**目的:** 特定テーマを深掘りし、読者に独自の価値を提供

**特徴:**
- 速報性より「深さ」と「独自性」を重視
- 既存リソースの評価・キュレーションを含む
- リサーチ段階で人間的判断が必要

| 観点 | 朝刊・夕刊Digest | 個別記事 |
|------|-----------------|---------|
| 目的 | 速報性・全体像把握 | 深さ・独自価値 |
| 時間軸 | 当日〜前日のニュース | タイムレス or 旬のテーマ |
| 記事長 | 3,000〜5,000字 | 8,000〜20,000字 |
| 読了時間 | 5〜8分 | 10〜20分 |
| 更新頻度 | 毎日2回 | 週2〜3本 |
| cron | asb-morning-digest / asb-evening-digest | asb-midday-editorial |
| 自動化度 | 高い（5 Phase パイプライン） | 中程度（リサーチは手動要素多い） |

---

## 個別記事の3つの型

### 1. キュレーション型（★推奨）
> 既存の良質なリソースを評価・比較し、読者に最適な学習パスを案内する

**特徴:**
- 一次ソースへの敬意を払いつつ、独自の評価軸で整理
- 「どれを読むべきか」「どの順番で学ぶべきか」を提示
- 原文リンク + 日本語での要点解説 + 独自の実践視点

**例:**
- 「Claude Code活用ガイド比較レビュー — 公式ドキュメント vs 海外記事 vs 実践者の知見」
- 「MCPサーバー開発の学習ロードマップ — 初心者から実装まで」

### 2. 事例分析型
> 実際の成功/失敗事例を深掘り分析し、再現可能な教訓を抽出

**特徴:**
- 具体的な数字（売上、ユーザー数、開発期間）
- 時系列での軌跡
- 「なぜうまくいったか/いかなかったか」の分析
- 読者が活かせる原則の明確化

**例:**
- 「Pieter Levelsの事例 — 従業員ゼロ・AIだけで年収3億円」
- 「Lovableの事例 — 2ヶ月でARR $4M到達の戦略」

### 3. 実践ガイド型
> 特定のスキル・ツールを「使えるようになる」ための具体的手順

**特徴:**
- 実際に動くコード・設定例
- ビフォー/アフターの比較
- よくある失敗とその回避法
- 段階的な難易度設定

**例:**
- 「Claude Codeで本番プロダクトを作る完全ガイド」
- 「MCPサーバー構築 — ゼロから動くサーバーまで」

## 5 Phase 構成

```
Phase 1          Phase 2            Phase 3          Phase 4          Phase 5
[テーマ選定] ──▶ [リサーチ] ──────▶ [評価] ──────▶ [設計] ──────▶ [執筆・公開]
   │                │                  │                │                │
   ▼                ▼                  ▼                ▼                ▼
テーマ決定      一次ソース収集      リソース評価      独自価値設計      記事公開
(30分)          (1-2時間)           (1時間)           (30分)           (2-4時間)
```

---

## Phase 詳細

### Phase 1: テーマ選定（30分）

| 項目 | 内容 |
|------|------|
| 責務 | 読者価値・差別化・ソース存在確認 |
| 入力 | テーマ候補（HN/X/GitHub Trending/PH/読者リクエスト） |
| 出力 | テーマ決定 |

**チェックリスト:**
- [ ] ターゲット読者（AIソロ開発者）にとって価値があるか
- [ ] 既存記事との差別化ポイントがあるか
- [ ] 一次ソースが十分に存在するか
- [ ] 自分（または参照可能な実践者）に実体験があるか

**テーマ発見ソース:**
- Hacker News（AI/dev関連で100+ポイント）
- X/Twitter（@levelsio, @karpathy, @simonw等）
- GitHub Trending
- ProductHunt
- 読者からの質問・リクエスト

### Phase 2: 一次ソースリサーチ（1〜2時間）

| 項目 | 内容 |
|------|------|
| 責務 | 公式ドキュメント・作者発言・海外記事・実践者報告収集 |
| 入力 | テーマ決定 |
| 出力 | 一次ソースを最低3つ確保 |

**収集対象（優先順）:**
1. **公式ドキュメント** — 最も信頼性が高い
2. **作者本人の発言** — ブログ、ポッドキャスト、X投稿
3. **海外の質の高い記事** — Entrepreneur Loop, TechCrunch, The Verge等
4. **実践者の報告** — GitHub Issues, Reddit, HN comments
5. **ベンチマーク・データ** — 定量的な根拠

**記録フォーマット:**
```markdown
## ソース一覧
| # | ソース名 | URL | 種類 | 信頼度 | 要点 |
|---|---------|-----|------|--------|------|
| 1 | 公式ドキュメント | ... | 公式 | ★★★ | ... |
| 2 | 作者のX投稿 | ... | 一次 | ★★★ | ... |
| 3 | 海外記事 | ... | 二次 | ★★☆ | ... |
```

### Phase 3: 既存リソース評価（1時間）

| 項目 | 内容 |
|------|------|
| 責務 | 各リソースを評価し、推奨・順番を決定 |
| 入力 | 収集済みソース |
| 出力 | 推奨リソースリスト + 強み/弱み分析 |

**評価軸:**
| 軸 | 説明 |
|----|------|
| 正確性 | 情報が正確か、最新か |
| 深さ | 表面的 vs 本質的 |
| 実用性 | 読んだ後に行動できるか |
| 独自性 | 他で読めない視点があるか |
| 日本語対応 | 日本語で読めるか、日本向け解説があるか |

### Phase 4: 独自価値の設計（30分）

| 項目 | 内容 |
|------|------|
| 責務 | 「この記事でしか得られない価値」を明確化 |
| 入力 | 評価済みリソース |
| 出力 | 記事の構造設計 |

**独自価値の出し方:**
1. **日本向けローカライズ** — 海外記事の日本市場への適用
2. **比較・評価** — 複数のリソースを横断的に評価
3. **実践検証** — 実際に試した結果を共有
4. **構造化** — 散らばった情報を体系的に整理
5. **学習パス提示** — 「次に何をすべきか」を明示

### Phase 5: 執筆・品質チェック・公開（2〜4時間）

| 項目 | 内容 |
|------|------|
| 責務 | 記事執筆 → content-optimizer → article-quality-check → publish-gate |
| 入力 | 構造設計 + ソース |
| 出力 | 本番サイト公開 |

**スキル連携:**
```
[手動リサーチ]
     │
     │ 一次ソース・参考記事リスト
     ▼
[記事執筆]
     │
     │ content/news/*.md (初稿)
     ▼
content-optimizer
     │
     │ content/news/*.md (UI最適化済み)
     ▼
article-quality-check
     │
     │ 品質基準クリア
     ▼
publish-gate
     │
     │ git push → Vercel deploy
     ▼
   公開完了
```

**必須要素:**
- [ ] 冒頭で「この記事で得られること」を明示
- [ ] 一次ソースへのリンク（敬意を払う）
- [ ] 具体的な数字・データ
- [ ] 表形式での比較（読みやすさ）
- [ ] 実際のコード例・設定例（該当する場合）
- [ ] 「次のアクション」を明示
- [ ] 関連プロダクトへのリンク

**品質チェック:**
- [ ] 8,000字以上（深掘り記事の場合）
- [ ] 読了時間10分以上
- [ ] 「どこでも読める内容」になっていないか
- [ ] 実証・データがあるか
- [ ] 読者が行動に移せるか

**完了条件:** デプロイ完了、PC/モバイル表示確認済み

## 良い個別記事の実例

### Pieter Levels記事
- 15,700字・12分
- 具体的な収益データ（月$138,000等）
- 時系列での軌跡
- 再現可能な5原則を抽出
- 日本のソロビルダー向けの視点

### Claude Codeビジネスアイデア記事
- 18,900字・15分
- 海外記事の紹介+独自分析
- 15アイデアの表形式一覧
- 各アイデアの日本向け評価

## スケジュール目安

| フェーズ | 時間 |
|---------|------|
| テーマ選定 | 30分 |
| 一次ソースリサーチ | 1〜2時間 |
| 既存リソース評価 | 1時間 |
| 独自価値設計 | 30分 |
| 執筆 | 2〜4時間 |
| 品質チェック・公開 | 30分 |
| **合計** | **5〜9時間** |

朝刊・夕刊の合間に、週2〜3本のペースで制作。

  - `contentType: "news"`
  - `tags: ["dev-knowledge"]` or `tags: ["case-study"]`
