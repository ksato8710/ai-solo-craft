# 朝刊統合仕様（ニュースソース連携ルール）

最終更新: 2026-02-20

この仕様は、複数ニュースソースを朝刊へ統合する際の「役割」「採用条件」「必要情報量」を定義する。

## 1. ソースの関係性（役割分離）

1. `検知ソース (detector)`
- 対象: 海外ニュースレター、速報メディア
- 役割: 話題の早期検知
- 禁止: これ単体で断定しない

2. `一次検証ソース (verifier)`
- 対象: 公式ブログ、公式リリース、企業発表、一次データ
- 役割: 事実確定（仕様・数値・日付）
- 必須: Top3は必ず1本以上

3. `日本語補足ソース (localizer)`
- 対象: 信頼できる国内メディア（ITmedia NEWS, 日経クロステック, Publickey 等）
- 役割: 国内文脈の補足、読み手が理解しやすい導線
- 必須: Top3は必ず1本以上

4. `補助比較ソース (benchmark)`
- 対象: グローバルメディア、アナリスト記事、競合情報
- 役割: 位置づけ比較（市場・競合）
- 任意: Top3で必要な場合のみ

## 2. 朝刊の構成と情報量基準

### 2-0. 現時点で設けない枠

- 広告枠
- コミュニティ投稿枠

注: 現フェーズでは速報品質の安定を優先し、上記枠は追加しない。

### 2-1. Top3（深掘り枠）

- 目的: 読者が「今日動くべきか」を判断できること
- 必須情報:
  - 概要（120〜220字）
  - なぜ重要か（80〜160字）
  - 主要ポイント3点（各40〜100字）
  - 公式発表（原文）リンク 1本以上
  - 日本語の解説記事リンク 1本以上
- 推奨:
  - 本サイト詳細解説リンク 1本
  - 今日の実行アクション 1〜3項目

### 2-2. 速報一覧（Quick Hits, #4-#10）

- 目的: 全体俯瞰と取りこぼし防止
- 掲載条件（すべて満たすこと）:
  - Top3と重複しない
  - `NVA >= 75` を満たす
  - 参照リンク（一次情報または信頼できる報道）を1本以上持つ
  - Solo Builderに対する影響が一言で説明できる
    - 例: API仕様変更、価格改定、実装導線の変化、配信/集客への直接影響
  - 噂・未確認・リンク切れ情報ではない
- 必須情報量:
  - 見出し（40〜90字）
  - NVAスコア
  - 参照リンク 1本以上
- 制約:
  - 1件あたり1行を基本とし、深掘りはしない
  - 速報一覧のための埋め草（意味の薄い低インパクト項目）を禁止

### 2-3. 今日の実行アクション

- 目的: 情報を行動に変換
- 必須:
  - 3項目以内
  - 1項目は具体的作業（例: 「モデル振り分けを設定」）
  - 抽象表現のみは禁止（例: 「注目する」だけ）

## 3. 採用判定ルール（朝刊統合前）

1. 速報性
- Top3候補: 原則24時間以内
- Quick Hits: 原則48時間以内

2. 根拠の強さ
- Top3候補は verifier 不在なら不採用
- detectorのみの話題は「観測情報」扱いで速報一覧へ降格
- Quick Hits候補もリンク検証に失敗した場合は不採用

3. Solo Builder関連性
- 実装・収益・運用のいずれかに直接影響があること
- 影響が説明できない話題は優先度を下げる

## 4. データモデル対応

- `sources`: 役割候補（`entity_kind`, `locale`, `is_newsletter`）
- `workflow_source_mappings`: 実運用での役割確定（detect/verify/localize/benchmark）
- `source_delivery_schedules`: detectorの到着時刻管理
- `content_workflows`: 朝刊の出力契約（Top3/Quick Hits/Action）

## 5. 運用チェックリスト（配信前）

- [ ] Top3各項目に「公式発表（原文）」リンクがある
- [ ] Top3各項目に「日本語の解説記事」リンクがある
- [ ] Top3各項目に概要・重要性・主要ポイントがある
- [ ] Quick Hitsは重複トピックを除去している
- [ ] 今日の実行アクションが3項目以内で具体的
- [ ] 他媒体の固有フォーマット語（例: "In today's rundown"）を残していない
